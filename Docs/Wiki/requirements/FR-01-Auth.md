## 1) Scope

### In Scope
- Password login for application roles (RESEARCHER, TEACHER, STUDENT) with role-based identifiers (student username; non-student username or email)
- OAuth login for code-gated non-student application accounts (RESEARCHER, TEACHER only)
- Token refresh (backend-only)
- Self-service password change
- Issuer-based password reset code flow for all roles (no SMTP, no request queue)
- Teacher-initiated student password reset (direct code generation from course roster)
- Researcher-initiated teacher password reset (direct code generation from teacher list)
- Admin-initiated reset code generation for any role
- Django admin login for ADMIN (system role)
- Logout

### Out of Scope
- Email-based password reset (SMTP)
- Self-registration (handled in FR-02 REG)
- First-time login / onboarding tokens (removed)

### Core Intent
- Provide secure, role-aware authentication for all platform users.
- Support issuer-based password reset without SMTP dependency.
- Enforce session security with non-enumerable error responses and rate limiting.

## 2) Actors

| Role | Type | Notes |
|------|------|-------|
| ADMIN | System role | `is_staff=True`; can use Django admin panel |
| RESEARCHER | User role | Highest user role; can generate teacher reset codes directly |
| TEACHER | User role | Can generate student reset codes directly from course roster |
| STUDENT | User role | Receives reset code from teacher; cannot self-issue reset codes |

**Actor ordering:** ADMIN > RESEARCHER > TEACHER > STUDENT

## 4) Use Cases

### Core Authentication

#### AUTH-UC-01 — Password Login

**Roles:** RESEARCHER, TEACHER, STUDENT

**Preconditions:** User account exists; account not disabled/suspended.

**Trigger:** User submits identifier + password on login page.

**Main Flow:**
1. User enters identifier + password.
2. System validates credentials.
3. On success, return user profile payload and set HttpOnly auth cookies.
4. User is redirected to dashboard.

**Postcondition:** Active session established.

**Role Coverage:**

> **AUTH-UC-01-RESEARCHER / TEACHER**
> - Identifier: Username or email
> - Entry: Application login page
>
> **AUTH-UC-01-STUDENT**
> - Identifier: Username generated at registration
> - Entry: Application login page

**Variants:**

> **AUTH-UC-01a** — Django Admin Login
>
> > **AUTH-UC-01a-ADMIN**
> > - Entry: `/admin/` portal
> > - Auth: Django admin authentication
> > - Scope: `is_staff=True` only
> > - Precondition: Admin account bootstrapped via environment (ENV-UC-02)

**Policy Gate:**
- ADMIN accounts MUST use Django admin (`/admin/`) and are rejected by `/api/v1/auth/sessions` with 403.

**Errors:**

**AUTH-UC-01-E1** — Generic login error
- Trigger: Invalid credentials
- Behavior: Generic error message; no user enumeration
- Constraint: AUTH-CN-04

**AUTH-UC-01-E2** — Rate limit error
- Trigger: Too many failed attempts on same identifier
- Behavior: Cooldown message; 5 attempts per 15 minutes
- Constraint: AUTH-CN-03

**AUTH-UC-01-E3** — Login blocked
- Trigger: Account disabled or suspended
- Behavior: Generic error (may be masked to prevent enumeration)
- Constraint: AUTH-CN-04

**Tests (representative):** test_AUTH_UC_01, test_AUTH_UC_01a_ADMIN, test_AUTH_UC_01_E1, test_AUTH_UC_01_E2, test_AUTH_UC_01_E3, test_AUTH_CN_04, test_AUTH_CN_12

**Role Coverage:**

**Errors:**

- Trigger: User cancels OAuth or provider error

**AUTH-UC-02-E2** — OAuth account not eligible
- Trigger: No prior code-gated registration, or account suspended
- Behavior: Error indicating account not found or not eligible

#### AUTH-UC-09 — Session Profile Lookup (`/auth/me`)

**Roles:** RESEARCHER, TEACHER, STUDENT

**Preconditions:** Access token is present and valid.

**Trigger:** Protected frontend layout loads and requests current profile.

**Main Flow:**
1. Frontend calls `GET /api/v1/auth/me`.
2. Backend validates session token and resolves canonical role from backend state.

**Postcondition:** Frontend role gating is based on backend source-of-truth.

**Errors:**

**AUTH-UC-09-E1** — Unauthenticated profile lookup
- Trigger: Missing/invalid token
- Behavior: 401 response; frontend redirects to login

### Password Management

#### AUTH-UC-04 — Change Password (self-service)

**Roles:** RESEARCHER, TEACHER, STUDENT

**Preconditions:** User is authenticated.

**Trigger:** User submits current password + new password + confirm.

**Main Flow:**
1. Verify current password.
2. Validate new password against strength policy (AUTH-CN-01).
3. Verify new password differs from old (AUTH-UC-04-E2).
4. Update password.
5. Invalidate outstanding refresh-token sessions (AUTH-CN-11).
6. Redirect to login.

**Postcondition:** Password updated; all sessions revoked; user must re-login.

**Role Coverage:**

> **AUTH-UC-04-RESEARCHER / TEACHER / STUDENT**
> - Behavior: Identical flow for application roles

**Errors:**

**AUTH-UC-04-E1** — Change password failed
- Trigger: Weak password, mismatch confirm, or incorrect current password
- Behavior: Specific error per failure type
- Constraint: AUTH-CN-01

**AUTH-UC-04-E2** — New password equals old
- Trigger: New password is identical to current password
- Behavior: Rejection with clear message

**Tests (representative):** test_AUTH_UC_04, test_AUTH_UC_04_E1, test_AUTH_UC_04_E2, test_AUTH_CN_01, test_AUTH_CN_11

### Issuer-Based Password Reset

#### AUTH-UC-05 — Password Reset (Code Consumption)

**Roles:** RESEARCHER, TEACHER, STUDENT

**Preconditions:** User has a valid account and a valid issued reset code (RESET-...).

**Main Flow:**
1. User receives reset code from authorized issuer via out-of-band channel.
2. User enters identifier + reset code.
3. System validates code ownership, expiry, and single-use status.
4. User sets new password and confirms.
5. System consumes code and updates password atomically.
6. No auto-login; user returns to login.

**Postcondition:** Password reset completed; code invalidated.

**Errors:**

**AUTH-UC-05-E1** — Reset code invalid/expired/used
- Trigger: Code cannot be redeemed (wrong code, time elapsed, already used)
- Behavior: Error message; user requests a new code from issuer
- Constraint: AUTH-CN-07, AUTH-CN-08

**AUTH-UC-05-E2** — Reset password invalid
- Trigger: New password is weak, mismatched confirm, or same as old
- Behavior: Validation error on set-new-password step
- Constraint: AUTH-CN-01

**Tests (representative):** test_AUTH_UC_05, test_AUTH_UC_05_E1, test_AUTH_UC_05_E2, test_AUTH_CN_06, test_AUTH_CN_07, test_AUTH_CN_08, test_AUTH_CN_09

---

#### AUTH-UC-06 — Reset Code Validation (Status Check)

**Roles:** RESEARCHER, TEACHER, STUDENT

**Preconditions:** User has identifier + issued reset code.

**Trigger:** User submits identifier + reset code before setting new password.

**Main Flow:**
1. User submits identifier + reset code.
2. System validates ownership, expiry window, and consumption status.
3. System returns validity response.

**Postcondition:** User knows whether code can be consumed by AUTH-UC-05.

**Errors:**

**AUTH-UC-06-E1** — Invalid reset code
- Trigger: Code does not exist or is not bound to identifier
- Behavior: Validation response indicates invalid

**AUTH-UC-06-E2** — Expired or consumed reset code
- Trigger: Code is outside TTL or already consumed
- Behavior: Validation response indicates invalid

**Tests (representative):** test_AUTH_UC_06, test_AUTH_UC_06_E2

---

#### AUTH-UC-07 — Reset Code Issuance (Issuer-Driven)

**Roles:** ADMIN, RESEARCHER, TEACHER

**Main Flow:**
1. Issuer opens allowed target list (course roster or staff list).
2. Issuer selects target account.
3. Issuer triggers reset code generation.
4. System validates issuer scope and target eligibility.
5. System generates one-time reset code with fixed profile (single-use, short TTL).
6. System returns code once; issuer shares out-of-band.

**Postcondition:** Target user has one active issued reset code.

**Reset Chain (Default + Sudo Extensions):**

| Target | Issuer | Trigger | Expiry Config |
|--------|--------|---------|---------------|
| Student | Teacher | Direct issuance from owned course roster | Fixed 30 min |
| Teacher | Researcher | Direct issuance from teacher list | Fixed 30 min |
| Researcher | Admin | Direct issuance from researcher list | Fixed 30 min |
| Student | Researcher (sudo) | Requires `ISSUE_STUDENT_RESET_CODE` | Fixed 30 min |
| Researcher | Researcher (sudo) | Requires `ISSUE_RESEARCHER_RESET_CODE` | Fixed 30 min |
| Any | Admin | Direct issuance | Fixed 30 min |

**Errors:**

**AUTH-UC-07-E1** — Issuance denied
- Trigger: Issuer lacks scope/permission for target role
- Behavior: 403 with generic authorization message

**AUTH-UC-07-E2** — Target not in issuer scope
- Trigger: Teacher selects non-enrolled student or unknown target
- Behavior: 400/404 based on validation path

**Tests (representative):** test_AUTH_UC_07, test_AUTH_UC_07_ADMIN, test_AUTH_UC_07_E1, test_AUTH_UC_07_E2

## 5) Constraints

### AUTH-CN-01 — Password Strength Policy
- Minimum 8 characters
- At least 1 uppercase letter
- At least 1 lowercase letter
- At least 1 number
- At least 1 special character
- **Applies to:** AUTH-UC-04, AUTH-UC-05 (set new password step)
- **Implements:** NFR-SEC-04 (Password Strength Policy) - (user password strength; bootstrap admin covered in ENV-CN-04)

### AUTH-CN-02 — JWT Token Lifetimes
- Access token lifetime: 15 minutes (`SIMPLE_JWT.ACCESS_TOKEN_LIFETIME`)
- Refresh token lifetime: 24 hours (`SIMPLE_JWT.REFRESH_TOKEN_LIFETIME`)
- Tokens are sent in HttpOnly cookies with matching `max_age`
- Refresh does not extend beyond the original 24h refresh token ceiling
- **Applies to:** AUTH-UC-01, AUTH-UC-02, AUTH-UC-03
- **Implements:** NFR-SEC-05 (Session Security)

### AUTH-CN-03 — Rate Limiting
- Per-identifier throttle: 5 failed attempts per 15 minutes per identifier
- Applies to all identifiers equally (existing and non-existing) to prevent enumeration
- No IP-based throttle (school NAT environment)
- Global request throttle as DDoS safety net (infrastructure-level)
- Reset-code verification/consumption endpoints use anonymous burst throttle, not identifier throttle
- **Applies to:** AUTH-UC-01, AUTH-UC-02
- **Implements:** NFR-SEC-01 (Rate Limiting Protection) - (per-identifier throttle mechanism; NFR specifies per-IP but school NAT environment requires role-aware identifier keys)

### AUTH-CN-04 — No User Enumeration
- Login failure returns generic error regardless of whether identifier exists
- Rate limiting behavior is identical for existing and non-existing identifiers
- Suspended/disabled accounts may be masked behind generic error
- **Applies to:** AUTH-UC-01, AUTH-UC-05
- **Implements:** NFR-SEC-03 (Enumeration Prevention)

### AUTH-CN-05 — No Email-Based Reset
- Email/SMTP password reset is not available until SMTP infrastructure exists
- Reset is issuer-based only; no request queue/status lookup endpoints
- Student resets are teacher-issued from owned course rosters
- Teacher resets are researcher-issued by default
- Researcher resets are admin-issued by default
- **Applies to:** AUTH-UC-05

### AUTH-CN-06 — Reset Code Expiry Rules
- Default expiry: 30 minutes for all roles
- Fixed profile for all issuer-generated reset codes (no UI expiry picker)
- **Applies to:** AUTH-UC-05, AUTH-UC-07

### AUTH-CN-07 — Reset Codes Single-Use
- Each reset code can be used exactly once
- Code is invalidated immediately upon successful password reset
- **Applies to:** AUTH-UC-05

### AUTH-CN-08 — Transactional Reset Operations
- Reset code issuance and consumption must be atomic
- No partial states: either the full operation succeeds or rolls back
- **Applies to:** AUTH-UC-05, AUTH-UC-07
- **Implements:** NFR-REL-01 (Transaction Atomicity for Multi-Record Operations)

### AUTH-CN-09 — Reset Codes Temporary
- Reset codes are not archived or stored long-term
- Expired/used codes are cleaned up (no audit trail for code values)
- **Applies to:** AUTH-UC-05, AUTH-UC-07

### AUTH-CN-10 — Issuer-Generated Code Behavior
- Reset codes are generated by authorized issuers for a selected target user
- Code is shown once to issuer and shared out-of-band
- Code is bound to the target account and cannot be reused for another user
- New issuance for the same target invalidates prior active reset code
- **Applies to:** AUTH-UC-07

### AUTH-CN-11 — Session Invalidation on Password Change
- Successful password change invalidates all outstanding refresh-token sessions
- Existing access tokens may remain valid until access-token expiry window closes
- User must log in again after access token expiry using the new password
- This applies to self-service change (AUTH-UC-04) only, not to reset (AUTH-UC-05)
- **Applies to:** AUTH-UC-04
- **Implements:** NFR-SEC-05 (Session Security) - (session invalidation on credential change)

### AUTH-CN-12 — Role-Based Login Identifiers
- STUDENT accounts authenticate with immutable username only
- RESEARCHER and TEACHER authenticate with username or email
- ADMIN authenticates through Django admin (`/admin/`) only
- UI may use a single `Identifier` field, but backend must enforce role-specific rules
- For AUTH-UC-05 reset code consumption, students use username; non-students use username or email
- **Applies to:** AUTH-UC-01, AUTH-UC-05

### AUTH-CN-13 — Student OAuth Disabled
- Google OAuth login is not available for STUDENT accounts
- OAuth endpoints must reject student-role mappings with a clear unsupported-flow error
- **Applies to:** AUTH-UC-02

### AUTH-CN-14 — Backend-Sourced Session Role
- Frontend role gating must use `GET /api/v1/auth/me` as source-of-truth
- Role cookies are not issued; access control must not depend on client-managed role state
- Invalid/unknown role values from `/auth/me` must force login redirect
- **Applies to:** AUTH-UC-09

#### Password Change

| Method | Path | Auth | UC |
|--------|------|------|----|
| PATCH | `/api/v1/auth/password` | Authenticated session | AUTH-UC-04 |

#### Reset Code Issuance (Issuer-Driven)

| Method | Path | Auth | UC |
|--------|------|------|----|
| POST | `/api/v1/auth/password-reset-codes` | Admin / Researcher / Teacher | AUTH-UC-07 |

**POST payload example:**
- `{ "targetUserId": 123 }`

Notes:
- Teacher issuance is limited to students enrolled in teacher-owned courses.
- Researcher default issuance target is TEACHER; student/researcher targets require sudo permission.
- Code profile is fixed: one-time use, 30-minute expiry, target-bound.

#### Reset Code Consumption (All Roles)

| Method | Path | Auth | UC |
|--------|------|------|----|
| POST | `/api/v1/auth/reset-code-validations` | None | AUTH-UC-06 |
| POST | `/api/v1/auth/password-resets` | None | AUTH-UC-05 |

Notes:
- `/api/v1/auth/sessions/oauth` is for RESEARCHER/TEACHER only; STUDENT attempts must be rejected per AUTH-CN-13 and ADMIN attempts must be rejected to enforce Django-admin-only policy.

#### Admin Panel

| Path | Auth | UC |
|------|------|----|
| `/admin/` | Admin | AUTH-UC-01a-ADMIN |

### 6.3 Reset Code States

Applies to all roles. Reset requests are removed; issuance always starts from an issuer action.

```
[Generated by issuer action] → Active → Used (password reset successful)
                                      → Expired (time elapsed)
```

## 7) Error Model

| Scenario | Behavior | Contract |
|----------|----------|----------|
| Invalid credentials (login) | Generic error; no user enumeration | `401` |
| Rate limit exceeded (login) | Cooldown message; 5 attempts per 15 min | `429` |
| Account disabled/suspended | Generic error (may be masked) | `401` |
| OAuth denied/failed | Clear error; return to login | `400` |
| OAuth account not eligible | Account not found/not eligible | `403` |
| Weak/mismatched password (change) | Field-level validation errors | `400` |
| New password equals old | Rejection with clear message | `400` |
| Reset code invalid/expired/used | Error; request new code from issuer | `400` |
| Reset password validation failure | Validation error on set-new-password | `400` |
| Issuance denied (no scope) | 403 authorization message | `403` |
| Target not in issuer scope | Validation/not-found error | `400`/`404` |
| Student OAuth attempt | Unsupported-flow error | `403` |
| Unauthenticated profile lookup | 401 response; frontend redirects to login | `401` |
| Admin API login attempt | Rejected; must use Django admin | `403` |

## 11) Current Implementation Alignment Notes

### Shared Code Generation

Auth reset codes use the same **core code generator** as registration codes, but with a **different policy profile**. This keeps generation logic consistent while preserving domain-specific behavior.

- **Registration policy (REG):** persistent storage, multi-use, revokable/archivable, course linkage
- **Reset policy (AUTH):** short-lived, single-use, target-bound, issuer-generated (no request-token queue)

This is an implementation detail; requirements remain scoped under AUTH and REG.
